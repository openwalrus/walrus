# 0080 - Cron

- Feature Name: Cron Scheduler
- Start Date: 2026-03-20
- Discussion: [#80](https://github.com/crabtalk/crabtalk/issues/80), [#183](https://github.com/crabtalk/crabtalk/issues/183)
- Crates: `wcore::trigger::cron`, `plugins/cron`

## Summary

Cron triggers skills into agents on a schedule. The scheduler runs as a standalone service outside the daemon and speaks the existing `StreamMsg` protocol ŌĆö no cron-specific daemon knowledge, no cron-specific wire messages. Shared data types (`CronEntry`, schedule validator, quiet-hours helper) live in `wcore::trigger::cron` so alternative consumers (e.g. multi-tenant schedulers backed by a database) reuse them without the TOML file format or the runner loop.

## Motivation

Agents need periodic behavior ŌĆö checking feeds, running maintenance, sending reminders. Time-based triggering is one form of *trigger*; chat messages, webhooks, and file-watch events are others. All of them produce the same shape: something happens ŌåÆ an agent runs with a payload. Cron is the first concrete implementation of this trigger role and deliberately uses the same `StreamMsg` path that chat gateways use.

The session already carries the agent and sender. A cron entry needs the skill to fire, the agent to run, the sender to attribute it to, and the schedule expression ŌĆö nothing else.

## Design

### Data model

```toml
[[cron]]
id = 1
schedule = "0 */2 * * * *"
skill = "check-feeds"
agent = "crab"
sender = "cron"
quiet_start = "23:00"
quiet_end = "07:00"
once = false
```

- `id` ŌĆö auto-incremented on create.
- `schedule` ŌĆö standard cron expression, validated on create and load.
- `skill` ŌĆö fired as `/{skill}` content into the target conversation.
- `agent` ŌĆö agent running the conversation.
- `sender` ŌĆö sender attribution (default `"cron"`).
- `quiet_start` / `quiet_end` ŌĆö optional `HH:MM` window in local time. If the fire time falls inside, the tick is skipped silently. No queuing, no catch-up. Both must be set; otherwise quiet hours are ignored.
- `once` ŌĆö fire once then delete.

### Architecture

Cron is a separate binary (`crabtalk-cron`). It runs as a system service managed by crabup / launchd / systemd. The daemon has no cron code ŌĆö no field, no handlers, no protocol messages.

The binary has three roles selected by subcommand:

- `crabtalk-cron run` ŌĆö the scheduler loop (invoked by the service unit).
- `crabtalk-cron start | stop | logs` ŌĆö install / uninstall / tail the system service.
- `crabtalk-cron create | list | delete` ŌĆö admin the schedule file directly.

### Persistence

The scheduler owns `~/.crabtalk/cron/crons.toml`. Admin subcommands read and write this file. The running service polls its mtime every 2 seconds; when it changes, the service reloads and reconciles timers ŌĆö abort removed schedules, start new ones. Atomic write (tmp + rename) keeps readers consistent.

For `once` schedules the service deletes the entry after firing.

### Firing

On a scheduled tick the service opens a `NodeClient` to the daemon and sends:

```rust
StreamMsg {
    agent: "<from entry>",
    content: "/<skill>",
    sender: Some("<from entry>"),
    ..Default::default()
}
```

The reply stream is drained and discarded ŌĆö output goes to conversation history through the daemon's normal path. Failures surface as `ErrorMsg` frames; the schedule continues on the next tick.

### Shared surface in `wcore::trigger::cron`

- `CronEntry` ŌĆö data shape with serde derives. Consumers pick their own storage (TOML file, database row, etc.).
- `validate_schedule(expr) -> Result<(), String>` ŌĆö cron expression parser.
- `is_quiet(start, end) -> bool` ŌĆö quiet-hours check on the current local time.

No file I/O, no TOML wrapper, no runner loop. Those are deployment concerns that live in the consuming binary.

## Alternatives

**Keep cron inside the daemon.** Rejected. Cron forced a `P: Provider` bound on the daemon struct for no reason other than that `CronStore` called `runtime.send_to`. Embedding also prevented alternate schedulers (e.g. multi-tenant cloud schedulers) from reusing the daemon without running cron.

**Introduce an `InvokeSkill` protocol variant.** Rejected. Cron already has everything it needs from `SendMsg` / `StreamMsg` ŌĆö the content `/{skill}` pattern is what chat-driven slash commands use too. Adding a variant would fragment the trigger contract across consumers and force every downstream scheduler (e.g. external ones) to learn a new wire format.

**Cron as a peer protocol endpoint with its own socket for admin.** Rejected. The admin surface is three operations; routing them through a dedicated socket multiplies client complexity (TUI would need to discover and connect to cron too). Instead, admin is a subcommand on the cron binary ŌĆö writes go directly to the schedule file, the running service picks them up via mtime polling.

**Introduce a `Trigger` trait / `crates/trigger` library.** Premature. Cron is the only trigger today (chat gateways are structurally similar but each has its own domain-specific code ŌĆö Telegram auth, WeChat sync, etc.). A common trait only becomes clear once a second non-chat trigger lands (webhook, file-watch). Until then, `wcore::trigger::cron` reserves the namespace without committing to an abstraction.

## Unresolved Questions

- Should cron support a time-zone override per entry (instead of local time for everything)?
- Should there be a max-concurrent-fires limit so a quiet window ending doesn't burst?
